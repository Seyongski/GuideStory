# docs/mechanics-harness.md — 메카닉 구현 하네스

> 게임 디자인([game-design.md](game-design.md))의 시스템을 코드로 옮기는 표준 절차.
> 모든 메카닉은 [../ARCHITECTURE.md](../ARCHITECTURE.md)의 ADR과 **불변 규칙(SDL 비의존)** 을 따른다.

## 1. 구현 절차

1. **디자인 근거** — game-design.md의 어느 시스템인가?
2. **상태 정의** — 읽고/쓰는 게임 상태.
3. **입력/출력** — 트리거와 효과.
4. **규칙** — 상태 전이 규칙.
5. **엣지 케이스** — 경계/실패/동시성(서버 권위 환경).
6. **검증 + 증명** — 테스트/플레이 + (해당 시) 벤치마크/분석.

## 2. 핵심 메카닉별 구현 노트

### 2.1 캐릭터 이동 & 점프 물리 (1단계)
- 상태: 위치(`Vector2D`), 속도, 접지 여부.
- 규칙: 중력 적용 → 발판 AABB와 충돌 시 착지, 점프 입력 시 상향 속도.
- 주의: 좌표/사각형은 **자체 `Vector2D`/`Rect`** 사용 (SDL_Rect 금지). → ADR-006

### 2.2 AABB 충돌 (2단계)
- 두 `Rect`의 교차 판정. 분리축(겹침) 계산.
- 충돌 응답: 발판 착지/벽 막힘/피격 판정.

### 2.3 QuadTree 공간 분할 (2단계)
- 정적/동적 객체 삽입, 영역 질의로 충돌 후보만 추림.
- **증명 과제(ADR-004)**: 브루트포스 경로를 함께 유지해 동일 씬에서 프레임타임(ms)·CPU를 비교. 동적 객체 재삽입 오버헤드를 한계로 기록.

### 2.4 몬스터 FSM (2단계)
- 상태: Idle / Patrol / Chase / Attack / Hit / Dead (예시).
- 전이: 거리/타이머/피격 이벤트 기반. 파라미터는 데이터로 주입.
- 상태별 행동은 컴포넌트로 분리 가능. → ADR-003

### 2.5 데이터 로드 (DataManager, 2단계)
- 시작 시 JSON/CSV 파싱 → 메모리 캐싱.
- **증명 과제(ADR-005)**: 데이터 오염 대비 예외 처리. 누락/형식 오류 시 안전 폴백 또는 명확한 실패.

### 2.6 네트워크 동기화 (3단계)
- 클라 입력 패킷 → 서버 판정 → 상태 브로드캐스트.
- 동시성: 멀티스레드 세션, 공유 상태 보호. → ADR-001

## 3. 메카닉 템플릿

```
### 메카닉: <이름>
- 디자인 근거: (game-design.md 섹션)
- 관련 ADR: (ARCHITECTURE.md)
- 모듈: (src/...)
- 상태:  읽기 / 쓰기
- 규칙:  WHEN <조건> THEN <효과>
- 엣지 케이스:
- 검증/증명: [ ] 테스트  [ ] (해당 시) 벤치마크/분석
```

## 4. 공통 규약

- 비즈니스 로직은 **SDL 네이티브 타입 비의존** (불변 규칙).
- 상태 변경은 권한 있는 시스템 한 곳에서만 (멀티플레이 시 서버).
- 결정론이 필요한 로직은 부동소수/난수에 주의.
- 메카닉은 렌더/입력과 분리해 테스트 가능하게.
