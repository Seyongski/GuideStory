# docs/product-specs.md — 기능 단위 상세 스펙

> [../PRODUCT.md](../PRODUCT.md) 비전을 구현 직전의 상세 요구사항으로 분해한다.
> 각 스펙은 [../ARCHITECTURE.md](../ARCHITECTURE.md)의 ADR과 [../PLANS.md](../PLANS.md)의 단계에 연결된다.

## 1. 스펙 작성 템플릿

```
### SPEC-<번호>: <기능명>
- 상태: 초안 | 확정 | 구현 중 | 완료
- 단계 / 관련 ADR:
#### 목적(Why)
#### 동작(How): 정상 흐름 / 대안 흐름
#### 수용 기준: [ ] GIVEN ~ WHEN ~ THEN ~
#### 증명 과제(있으면): 산출물
#### 범위 밖 / 의존성 / 위험
```

### SPEC-001: 플랫폼 추상화 + 빈 윈도우
- 상태: 초안 / 1단계 / ADR-006
- **목적**: SDL2를 `RenderDevice`/`Window` 인터페이스 뒤로 숨겨, 비즈니스 로직이 SDL에 비의존하도록.
- **동작**: 엔진이 `Window` 인터페이스로 창 생성·클리어·present. 내부 구현만 SDL2 호출.
- **수용 기준**:
  - [ ] 게임 로직 코드에 `SDL_*` 타입이 등장하지 않는다.
  - [ ] 자체 `Vector2D`/`Rect`로 좌표/영역을 표현한다.
  - [ ] 빈 윈도우가 뜨고 정상 종료된다(RAII로 자원 해제).
- **증명 과제(ADR-006)**: 직접 결합 vs 추상화 레이어의 이식성/복잡성 트레이드오프 보고.

### SPEC-002: RAII 리소스 매니저
- 상태: 초안 / 1단계 / ADR-002
- **목적**: SDL 핸들 수명을 RAII로 안전 관리.
- **동작**: `WindowManager`/`TextureManager`가 Custom Deleter 스마트 포인터로 SDL 핸들 보유.
- **수용 기준**:
  - [ ] 명시적 `SDL_Destroy*` 호출이 매니저 외부에 없다.
  - [ ] 예외/조기 반환에도 누수가 없다.
- **증명 과제(ADR-002)**: Custom Deleter 적용 과정·안정성 회고.

### SPEC-003: DataManager
- 상태: 초안 / 2단계 / ADR-005
- **목적**: 스킬·몬스터·아이템 수치를 외부 JSON/CSV로 분리, 시작 시 캐싱.
- **동작**: 시작 시 데이터 디렉터리 파싱 → 인메모리 테이블. 런타임 조회는 캐시에서.
- **수용 기준**:
  - [ ] 스킬 필드(쿨타임/타격횟수/범위/데미지계수/이펙트프레임) 로드.
  - [ ] 몬스터 스펙 로드.
  - [ ] 형식 오류/누락 시 예외 처리로 크래시 없이 진단 메시지.
- **증명 과제(ADR-005)**: 파싱 예외 처리 장단점, 로딩 시간 영향 기록.

### SPEC-004: QuadTree 충돌 + 벤치마크
- 상태: 초안 / 2단계 / ADR-004
- **목적**: 대량 객체 충돌 성능 확보 및 정량 증명.
- **수용 기준**:
  - [ ] 동일 씬에서 브루트포스/쿼드트리 전환 가능.
  - [ ] 스트레스 테스트(대량 몬스터+드롭)에서 프레임타임(ms)·CPU 측정.
- **증명 과제(ADR-004)**: 비교 데이터 + 동적 갱신 오버헤드 한계 명시.

### SPEC-005: 데디케이트 서버 패킷 구조
- 상태: 초안 / 3단계 / ADR-001
- **목적**: 서버 권위 멀티플레이 기반.
- **수용 기준**:
  - [ ] WinSock 패킷 구조체 정의(헤더/타입/페이로드).
  - [ ] 클라 입력→서버 판정→브로드캐스트 왕복.
  - [ ] 멀티스레드 서버에서 N명 동기화 검증.
- **증명 과제(ADR-001)**: IOCP vs Select 비교, 동기화 검증 로그.
