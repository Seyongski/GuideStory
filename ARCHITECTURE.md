# ARCHITECTURE.md — 시스템 아키텍처 & 설계 결정(ADR)

> 이 프로젝트는 "구현"이 아니라 **"설계 결정의 근거·트레이드오프·한계 증명"** 이 산출물이다.
> 따라서 이 문서의 핵심은 §4 **ADR(아키텍처 결정 기록)** 와 §5 **증명 과제 대시보드**다.
> 코드를 바꾸기 전 관련 ADR을 읽고, 결정을 바꾸면 ADR을 갱신한다.

## 1. 기술 스택

| 영역 | 선택 | 비고 |
|------|------|------|
| IDE / 툴셋 | Visual Studio 2022 (v143) | `GuideStory.sln` |
| 언어 표준 | **C++20** | ⚠️ vcxproj에 `LanguageStandard` 미설정 — [tech-debt-tracker.md](tech-debt-tracker.md) D-002 |
| 플랫폼 | Win32 / x64 (Console) | 서버·AI는 x64 권장 |
| 클라이언트 그래픽 | SDL2 | 추상화 레이어 뒤로 격리 (ADR-006) |
| 네트워크 | WinSock (IOCP / Select) | 데디케이트 서버 (ADR-001) |
| AI 학습/추론 | Python(학습) + libtorch(C++ 추론) | ADR-007 |
| C++ ↔ Python | ZMQ 또는 Socket | ADR-007 |
| 데이터 포맷 | JSON / CSV | 데이터 주도 (ADR-005) |
| 메모리 관리 | 스마트 포인터 + RAII | ADR-002 |

## 2. 시스템 토폴로지

```
┌────────────────────┐        패킷(입력/요청)        ┌──────────────────────────┐
│   클라이언트 (SDL2)  │ ───────────────────────────▶ │   데디케이트 서버 (WinSock) │
│  - 입력 수집         │                              │  - 지형/충돌 권위 계산      │
│  - 렌더 / 보간       │ ◀─────────────────────────── │  - FSM/스킬 판정           │
│  - 예측(선택)        │        브로드캐스트(결과)      │  - 멀티스레드 세션 관리      │
└────────────────────┘                              └────────────┬─────────────┘
                                                                  │ ZMQ/Socket
                                                                  ▼
                                                       ┌──────────────────────┐
                                                       │  AI 서브시스템         │
                                                       │  학습: Python          │
                                                       │  추론: libtorch(C++)   │
                                                       └──────────────────────┘
```

**서버 권위(Server-Authoritative)**: 플레이어가 이동/스킬을 쓰면 클라이언트가 즉시 적용하지 않고,
서버에 패킷을 보낸다. 서버가 지형·충돌을 계산한 뒤 결과를 모든 클라이언트에 브로드캐스트한다.

      ├─ math/        # Vector2D, Rect (SDL 비의존 — ADR-006)

|------|------|----------|----------|
| core | 메인 루프·상태 | ✗ | |
| math | Vector2D/Rect | ✗ (자체 정의) | ADR-006 |
| platform | 렌더/윈도우 추상화 | ✓ (격리됨) | ADR-006, ADR-002 |
| ecs | 컴포넌트 오브젝트 | ✗ | ADR-003 |
| physics | 충돌·공간분할 | ✗ | ADR-004 |
| ai | FSM·추론 | ✗ | ADR-007 |
| data | 데이터 로드 | ✗ | ADR-005 |

## 4. 아키텍처 결정 기록 (ADR)

> 각 ADR은 **결정 / 이유 / 트레이드오프 / 증명 과제(지침 지표)** 를 가진다.
> 증명 과제는 §5 대시보드에서 진행 상황을 추적한다.

### ADR-001 — 데디케이트 서버 (서버 권위 구조)
- **결정**: 클라이언트는 입력만 전송, 서버가 지형·충돌을 판정 후 브로드캐스트.
- **이유**: 치팅 방지, 단일 진실 원천, 멀티플레이어 동기화의 기반.
- **트레이드오프**: 입력 지연(RTT)↑ → 클라 예측/보간 필요. 서버 연산 부하↑.
- **증명 과제**: WinSock IOCP/Select 모델 비교, 멀티스레드 서버에서 N명이 한 맵에서 동기화되는지 검증. 패킷 구조체 설계 근거 기록.

### ADR-002 — RAII + 스마트 포인터 (자원 수명 관리)
- **결정**: SDL의 C 스타일 할당/해제를 C++ 클래스 생성자/소멸자(RAII)로 캡슐화. `TextureManager`, `WindowManager`는 **Custom Deleter** 를 가진 스마트 포인터로 SDL 핸들 관리.
- **이유**: 누수·이중 해제 방지, 예외 안전성.
- **트레이드오프**: 래퍼 작성 비용, 약간의 간접화.
- **증명 과제**: `SDL_DestroyTexture`/`SDL_DestroyWindow` 등을 `unique_ptr` Custom Deleter로 감싼 과정과, 그로 인해 확보한 안정성을 코드/문서로 기록.

### ADR-003 — 컴포넌트 기반 게임 오브젝트
- **결정**: 초기에는 가벼운 상속으로 시작하되, 스킬·몬스터 시스템 추가 시점에 **컴포넌트 패턴** 도입.
- **이유**: 상속 폭발(diamond/조합 폭증) 회피, 기획 변경 유연성.
- **트레이드오프**: 초기 설계 비용, 컴포넌트 간 통신 복잡성·캐시 지역성 고려.
- **증명 과제**: 상속→컴포넌트 전환 시점과 동기를 기록. 예) "플레이어 특정 스킬의 상태이상 구현" 같은 기획 변경에 컴포넌트 구조가 어떻게 유연히 대처했는지 **장단점 비교 기술**.

### ADR-004 — QuadTree 공간 분할 (충돌 최적화)
- **결정**: 충돌 후보 탐색을 Brute-Force(이중 루프) 대신 **QuadTree** 로.
- **이유**: 오브젝트 수 증가 시 O(n²) 프레임 드랍 방지.
- **트레이드오프**: 동적 객체 갱신(재삽입) 오버헤드, 트리 구축 비용.
- **증명 과제**: 대량 몬스터·드롭 아이템 **스트레스 테스트**. 브루트포스 vs 쿼드트리의 **프레임타임(ms)·CPU 점유율**을 데이터로 비교, 도입 배경과 **한계(동적 갱신 오버헤드)** 명시.

### ADR-005 — 데이터 주도 설계 (JSON/CSV + DataManager)
- **결정**: 스킬(쿨타임/타격횟수/범위/데미지계수/이펙트 프레임), 몬스터 스펙, 아이템을 코드가 아닌 외부 JSON/CSV로 분리. 엔진 시작 시 `DataManager` 가 메모리에 캐싱.
- **이유**: 기획자가 코드 수정 없이 수치 조절 → 유지보수성 극대화.
- **트레이드오프**: 파일 I/O·문자열 파싱으로 초기 로딩 시간↑, 데이터 포맷 오류 시 예외 처리 난이도↑.
- **증명 과제**: 파싱 중 데이터 오염을 막는 **예외 처리** 코드 포함, 그 장단점 기록.

### ADR-006 — SDL 디커플링 (이식성 / 결합도)
- **결정**: 자체 `Vector2D`, `Rect` 구조체 정의. 렌더링을 엔진 자체 `RenderDevice`/`Window` **인터페이스 뒤로 은닉**하고 내부에서만 SDL2 호출.
- **이유**: 미래 SDL3 / DirectX / Custom Graphics API 이식 시, 인터페이스 구현체만 교체.
- **트레이드오프**: 초기 설계 비용↑, 우회 호출로 미세한 구조적 복잡성↑.
- **증명 과제**: 비즈니스 로직이 SDL 네이티브 타입에 의존하지 않음을 보장. "SDL2→SDL3 이식 시 내 코드가 얼마나 유연한가(결합도)"를 **직접 결합 vs 추상화 레이어**로 비교·분석.

### ADR-007 — C++ 엔진 ↔ AI 서브시스템 통합
- **결정**: 학습은 Python(ML 에코시스템), C++ 엔진과는 ZMQ 또는 일반 Socket으로 통신. 학습된 모델은 추후 **libtorch** 로 C++ 엔진 내부에 직접 로드해 추론.
- **이유**: 학습 생산성(Python)과 런타임 성능/배포(C++) 분리.
- **트레이드오프**: 프로세스 간 통신 지연·직렬화 비용, 빌드 복잡성(libtorch 링크).
- **증명 과제**: 통신 레이어 지연 측정, 오프라인 학습→온라인 추론 전환 과정 기록.

> 이 프로젝트 "성공"의 척도. 각 항목은 코드 + **분석 문서/벤치마크**가 모두 있어야 완료.

| ID | 증명 과제 | 산출물 | 상태 |
|----|-----------|--------|------|
| ADR-001 | IOCP/Select 비교 + N명 동기화 검증 | 비교표, 검증 로그 | ☐ 미착수 |
| ADR-002 | Custom Deleter RAII 래핑 기록 | 코드 + 회고 | ☐ 미착수 |
| ADR-003 | 상속→컴포넌트 전환 장단점 비교 | 비교 문서 | ☐ 미착수 |
| ADR-004 | 브루트포스 vs QuadTree 벤치마크 | ms/CPU 데이터, 한계 분석 | ☐ 미착수 |
| ADR-005 | 파싱 예외 처리 + 장단점 | 코드 + 회고 | ☐ 미착수 |
| ADR-006 | SDL 디커플링 결합도 분석 | 결합/디커플 비교 보고 | ☐ 미착수 |
